Making Codex Observable: Sending Codex Telemetry to OpenTelemetry, Loki, and Grafana

AI coding agents are becoming part of daily engineering workflows. They read files, suggest changes, run commands, edit code, and sometimes interact with external systems. That makes observability important.

For traditional software, we usually ask:

  • What ran?
  • When did it run?
  • Did it succeed?
  • What changed?
  • Who approved it?
  • What output did it produce?

The same questions now apply to AI coding agents.

This post walks through a practical local setup for sending Codex Desktop telemetry into OpenTelemetry Collector, storing it in Grafana Loki, and viewing it in Grafana. It also explains what we actually observed in the logs, what did not show up, and how to think about the security implications.

The setup and observations in this post are based on hands-on testing and the official OpenAI Codex documentation, especially the Codex advanced configuration documentation and the Codex configuration reference.

Architecture

The local telemetry flow is simple:

Codex Desktop
  -> OpenTelemetry Collector
  -> Grafana Loki
  -> Grafana

Each component has a clear role:

  • Codex Desktop emits telemetry events.
  • OpenTelemetry Collector receives Codex telemetry over OTLP HTTP.
  • Loki stores the logs.
  • Grafana provides the UI to search and inspect events.

This is useful for local testing, proof-of-concept work, and understanding what Codex emits before sending telemetry to a production SIEM.

Why This Matters

Codex can perform tool-mediated actions such as:

  • Running shell commands
  • Editing files
  • Applying patches
  • Reading local workspace context
  • Calling local helper tools
  • Using MCP or app-provided capabilities

Without telemetry, these actions are hard to audit after the fact.

With OpenTelemetry enabled, we can start answering questions like:

  • Which tool did Codex call?
  • What arguments were passed?
  • Was the action approved automatically or manually?
  • Did the tool succeed or fail?
  • What output did the tool return?
  • Which conversation generated the event?

That is a meaningful step toward AI agent governance.

Local Docker Stack

The local stack uses three main containers:

otel-collector
loki
grafana

We also added a small one-time helper container:

loki-init

The helper exists because recent Loki containers run as a non-root user. On Windows/Docker Desktop, the mounted Loki volume may initially be owned by a user that Loki cannot write as. The symptom looked like this:

mkdir /tmp/loki/rules: permission denied

The fix was to create the required Loki directories and set ownership before Loki starts:

loki-init:
  image: alpine:3.20
  command: sh -c "mkdir -p /tmp/loki/chunks /tmp/loki/rules && chown -R 10001:10001 /tmp/loki"
  volumes:
    - loki-data:/tmp/loki
  restart: "no"

Loki then depends on this one-time helper:

loki:
  image: grafana/loki:3.7.2
  depends_on:
    loki-init:
      condition: service_completed_successfully

This allows Loki to keep running as its normal non-root user instead of weakening the setup by running Loki as root.

Codex Telemetry Configuration

Codex was configured to send telemetry to the local OpenTelemetry Collector endpoint:

[otel]
log_user_prompt = true
environment = "local"

[otel.exporter.otlp-http]
endpoint = "http://localhost:14318/v1/logs"
protocol = "binary"

The important settings are:

  • environment = "local" tags telemetry as local environment data.
  • log_user_prompt = true allows prompt text to be included in telemetry.
  • endpoint = "http://localhost:14318/v1/logs" points Codex to the local Collector.
  • protocol = "binary" uses OTLP HTTP binary encoding.

After changing Codex configuration, restart Codex so it loads the telemetry settings.

Reference: OpenAI documents Codex observability settings under the official Codex advanced configuration documentation. For broader configuration options, see the Codex configuration reference.

OpenTelemetry Collector Configuration

The Collector receives logs from Codex and forwards them to Loki:

receivers:
  otlp:
    protocols:
      http:
        endpoint: 0.0.0.0:4318
      grpc:
        endpoint: 0.0.0.0:4317

processors:
  resource:
    attributes:
      - key: service.name
        value: codex
        action: upsert
      - key: deployment.environment.name
        value: local
        action: upsert
  batch:

exporters:
  otlp_http/loki:
    endpoint: http://loki:3100/otlp
  debug:
    verbosity: basic

service:
  pipelines:
    logs:
      receivers: [otlp]
      processors: [resource, batch]
      exporters: [otlp_http/loki, debug]

The debug exporter is useful during testing because it shows whether the Collector is receiving log records.

When things are working, Collector logs show messages like:

Everything is ready. Begin running and processing data.

and later:

log records

Viewing Logs In Grafana

Open Grafana:

http://localhost:3000

Go to:

Explore -> Loki

Start with this basic query:

{service_name="codex"}

That confirms Codex logs are reaching Loki.

For tool results:

{service_name="codex"} | event_name="codex.tool_result"

For approval decisions:

{service_name="codex"} | event_name="codex.tool_decision"

To make tool results readable:

{service_name="codex"} | event_name="codex.tool_result"
| line_format "{{.event_timestamp}} tool={{.tool_name}} success={{.success}} args={{.arguments}} output={{.output}}"

To make approval decisions readable:

{service_name="codex"} | event_name="codex.tool_decision"
| line_format "{{.event_timestamp}} tool={{.tool_name}} decision={{.decision}} source={{.source}}"

One important detail: many useful values are stored as Loki structured metadata. The visible log body may look empty or minimal until you expand the log row or use line_format.

What We Observed

The Codex logs included events such as:

codex.websocket_event
codex.sse_event
codex.tool_decision
codex.tool_result

These event names align with the OpenAI Codex observability documentation, which describes Codex OpenTelemetry log events for conversations, API requests, streaming events, user prompts, tool decisions, and tool results.

Streaming response events looked like:

codex.websocket_event response.output_text.delta

These represent chunks of assistant response text being streamed to the UI.

The most useful audit events were:

codex.tool_decision
codex.tool_result

For shell commands, codex.tool_result included rich details such as:

  • Tool name
  • Command arguments
  • Working directory
  • Timeout
  • Sandbox or escalation settings
  • Approval justification
  • Success or failure
  • Duration
  • Output

For example, a harmless test command such as:

Get-Date

appeared in telemetry with the command, working directory, success status, and output.

Tools Observed In Loki

During testing, these tool names appeared under codex.tool_result:

apply_patch
shell_command
codex_appload_workspace_dependencies
codex_appread_thread_terminal
get_goal
list_mcp_resource_templates
list_mcp_resources
mcp__node_repljs
tool_search

These appeared under codex.tool_decision:

apply_patch
shell_command

This means Codex telemetry captured detailed results for several local tools, but approval decision events were most visible for actions such as shell commands and file patches.

What Did Not Appear

We also tested hosted web/search-style activity.

Searches for terms like:

web.run
search_query
example.com
devlino.com

did not appear in the local Loki telemetry stream.

That is an important finding.

It means this local Codex OpenTelemetry setup should not be treated as a full browser history or full network access log. It gives strong visibility into many tool-mediated local actions, but not every hosted or web-search action appeared in the local OTel data.

What Codex OTel Is Good For

Based on the testing, Codex OTel is useful for:

  • Auditing local shell commands
  • Reviewing file patch actions
  • Seeing tool names and arguments
  • Seeing tool success or failure
  • Capturing tool output snippets or output fields
  • Seeing approval decisions for certain tools
  • Correlating activity to a conversation
  • Tracking prompt activity when prompt logging is enabled

This is valuable for engineering governance and security review.

What It Is Not

Codex OTel should not be treated as complete forensic coverage.

It does not guarantee visibility into:

  • Full internal model reasoning
  • Every website visited
  • Every network request
  • Every byte of every file read
  • Full memory contents or memory change history
  • OS-level process activity outside Codex tools
  • Network activity outside Codex-controlled paths

Some of these may be visible when they happen through logged tools. For example, a shell command and its output may appear in codex.tool_result. But that does not mean every internal or hosted action is guaranteed to be logged.

Security Implications

The biggest security lesson is this:

Codex telemetry can contain sensitive data.

If Codex runs a command that prints a secret, the secret may appear in Loki.

For example:

Get-Content .env

could cause environment secrets to be stored in telemetry output.

Similarly, patch events may include file content, and shell command arguments may include paths, usernames, project names, or operational details.

Treat Loki and Grafana as sensitive systems.

Recommended controls:

  • Keep Loki, Grafana, and the Collector bound to 127.0.0.1 for local testing.
  • Do not expose Grafana publicly without authentication.
  • Change default passwords.
  • Add retention limits.
  • Add secret redaction if sending logs to a shared platform.
  • Avoid logging prompts in shared environments unless approved.
  • Route production telemetry to an enterprise SIEM with access control.
  • Combine Codex telemetry with DNS, proxy, firewall, EDR, and file integrity logs for deeper coverage.

Useful Queries

All Codex logs:

{service_name="codex"}

Tool results:

{service_name="codex"} | event_name="codex.tool_result"

Tool decisions:

{service_name="codex"} | event_name="codex.tool_decision"

Readable shell command results:

{service_name="codex"} | event_name="codex.tool_result" | tool_name="shell_command"
| line_format "{{.event_timestamp}} success={{.success}} args={{.arguments}} output={{.output}}"

Readable patch results:

{service_name="codex"} | event_name="codex.tool_result" | tool_name="apply_patch"
| line_format "{{.event_timestamp}} success={{.success}} patch={{.arguments}} output={{.output}}"

Search for web-related traces:

{service_name="codex"} |~ "(?i)(web.run|search_query|http|https|domain|url)"

Search for failed tools:

{service_name="codex"} | event_name="codex.tool_result" | success="false"
| line_format "{{.event_timestamp}} tool={{.tool_name}} args={{.arguments}} output={{.output}}"

Operational Lessons

Here are the main takeaways from the setup:

  1. Codex OTel works well for local tool auditing.
    Shell commands and file patches were clearly visible in Loki.
  2. Tool telemetry can be detailed.
    We saw command arguments, working directories, outputs, and approval context.
  3. Structured metadata matters.
    Grafana may show blank-looking log bodies unless you expand the row or use line_format.
  4. Web/search activity may not appear.
    Hosted web/search activity did not show up in our local Loki stream during testing.
  5. Telemetry is sensitive.
    Logs can contain command output, file paths, patches, and possibly secrets.
  6. This is observability, not total control.
    Codex OTel is one part of a broader security monitoring strategy.

Recommended Production Pattern

For a serious organizational rollout, I would not stop at local Loki.

A stronger architecture is:

Codex
  -> OpenTelemetry Collector
  -> Enterprise SIEM or log platform
  -> Alerting and investigation workflows

Add supporting controls:

DNS logs
Proxy logs
Firewall logs
EDR telemetry
Git audit logs
File integrity monitoring
Secret scanning
Access reviews
Retention policies

For example:

  • Use Codex OTel to see what tool was called.
  • Use EDR to see what process actually ran.
  • Use DNS/proxy logs to see what domains were reached.
  • Use Git/file monitoring to see what changed.
  • Use SIEM correlation to combine all of it.

That gives a much better audit story than any one source alone.

Final Thoughts

Codex OpenTelemetry gives teams a practical starting point for AI coding agent observability.

It helps answer:

What did the agent do?
Which tool was used?
Was it approved?
Did it succeed?
What output came back?

But it should not be mistaken for a complete forensic recorder.

The best way to think about it is:

Codex OTel gives strong visibility into supported Codex events and many tool-mediated actions.
For complete security monitoring, combine it with endpoint, network, identity, and repository telemetry.

That balance makes it useful without overestimating what it can prove.
